# OKR Navigator — 產品需求文件（PRD）

**版本**：v2.0  
**更新日期**：2026-04-18（依據 Roadmap v2 及產品方向討論更新）  
**產品名稱**：OKR Navigator  
**技術堆疊**：Next.js 16 · React 19 · TypeScript · Supabase · Tailwind CSS · Anthropic Claude API

---

## 1. 問題陳述（Problem Statement）

有想法的人往往不缺行動力，缺的是**清晰感**——不知道手上這件事現在做值不值得、對自己最重要的目標有沒有幫助。現有的個人目標管理工具（Notion、Todoist）都假設使用者已經知道該做什麼，只提供記錄與追蹤；而 OKR 工具的學習門檻又讓大多數人望而卻步。

OKR Navigator 要解決的是「**因目標模糊導致的行動癱瘓**」：使用者就算沒有目標管理的概念，只要有想做的事，就能透過 AI 把模糊的想法結構化為清晰的目標，並在每次打開 App 時，30 秒內知道現在最值得投資的一件事是什麼、以及為什麼。

若不解決這個問題，使用者將繼續在「有很多想法但不知道先做哪個」的狀態中消耗動力，目標管理工具也將淪為另一個沒人打開的 App。

---

## 2. 目標（Goals）

1. **30 秒決策**：使用者打開 App，在 30 秒內能看到所有 Idea 按加權貢獻分數排序，並展開看到每個 Idea 的一句理由，知道現在最值得做什麼。
2. **一句話輸入即有回饋**：使用者輸入 Idea 標題後，不需填寫額外欄位就能立刻看到初步評分與理由，其餘欄位全部選填。
3. **分數有脈絡、不只是數字**：每個 Idea 的加權分數反映目標優先級係數（用戶設定 1–3 級），理由說明為何這個時機做這件事效益最高。
4. **降低 OKR 建立門檻**：不懂 OKR 的使用者也能在 5 分鐘內建立第一個有意義的 Objective + SMART KR。
5. **核心循環閉環**：Idea → Task → KR 進度更新，三個環節無摩擦地串連，讓使用者感覺每個行動都在推進目標。

---

## 3. 非目標（Non-Goals）

1. **Deadline 不參與分數計算**：Idea 優先分數只考慮目標貢獻分與目標優先級係數，不加入 deadline 因素——用戶的截止日期變化性太大，強制納入反而降低分數可信度。
2. **不做最大化用戶在 App 內的使用時長**：產品定位是「30 秒得到指引，出去做事」，所有讓用戶在 App 裡待更久的功能都是低優先級；成功是用戶快速離開去執行，而不是滯留瀏覽。
3. **不做複雜的目標權重拖拉介面**：目標優先級使用 1–3 三級制，操作一個欄位即可，不做拖拉排序或百分比分配，降低設定門檻。
4. **AI 教練不在目標設定流程開頭出現**：引導式問答只在使用者輸入模糊 Idea 時介入，不在 OKR 設定頁的最開始放問卷流程——那個時機用戶動機低、摩擦感最強。
5. **不做團隊協作、行事曆整合、離線模式**：v2 前不考慮，原因同 v1 PRD。

## 4. 使用者故事（User Stories）

### Persona A：有想法但不確定優先順序的個人工作者

> 手上有 5–10 個想做的事，但每次打開任務清單都不知道從哪裡開始，最後什麼都沒做。

- 作為一位個人工作者，我想要打開 App 立刻看到所有 Idea 按貢獻分數排好序，這樣我就不需要自己判斷今天該先做哪件事。
- 作為一位個人工作者，我想要點開分數旁邊的展開鍵，看到一句話說明「為什麼這件事現在排第一」，這樣我才能信任這個排序並根據它行動。
- 作為一位個人工作者，我想要輸入一句話 Idea 就立刻看到初步評分，不用先填完所有欄位，這樣我有靈感時可以快速記錄而不被流程打斷。
- 作為一位個人工作者，我想要替每個目標設定優先級（1–3），這樣評分會反映我現在最在意的事，而不是對所有目標一視同仁。

### Persona B：目標模糊、剛開始做目標管理的使用者

> 對 OKR 完全陌生，但對生活有想法，希望有人幫他把模糊的方向變成可執行的計畫。

- 作為一位初學者，我想要輸入一句話目標描述就讓 AI 幫我產出 SMART KR，這樣我不需要知道什麼是 OKR 也能開始使用。
- 作為一位初學者，我輸入一個模糊的 Idea（如「提升自己」）時，我希望 AI 問我一個關鍵問題幫我釐清方向，這樣評分結果會更貼近我真實的意圖。
- 作為一位初學者，我想要在看到 Idea 分析理由後，理解「為什麼這件事對我的目標有幫助」，這樣我才能建立對工具的信任，持續使用它。

### Persona C：已有目標、需要持續追蹤的用戶

- 作為一位持續使用者，我希望完成一個 Task 後，對應 KR 的進度條即時更新，這樣我能感受到行動帶來的進展，維持動力。
- 作為一位持續使用者，我希望當我的某個目標長時間沒有進展時，相關 Idea 的評分理由會反映這個狀態，提醒我這個目標需要關注。

### Edge Cases

- 使用者尚未建立任何 OKR 就嘗試新增 Idea：顯示引導提示，說明需先建立至少一個目標，提供快速入口。
- 使用者所有目標的優先級都設為相同（全為 2）：系統正常計算，等同於不加權的簡單平均，不出錯。
- Claude API 失敗：Idea 可以手動填入分數或暫不評分儲存；OKR 建立流程的 AI 輔助欄位顯示「AI 暫時無法使用，請手動填寫」，不阻擋流程。
- Idea 輸入後 AI 判斷標題過於模糊（與任何 KR 相關性均 <3）：系統詢問一個釐清問題，而非直接輸出低分結果。

## 5. 功能需求（Requirements）

### Must-Have — P0

**5.1 使用者認證**（同 v1，無變更）
- 支援 Email + 密碼登入與註冊（Supabase Auth）
- 未登入時重導向至 `/auth`
- **驗收標準**：新用戶完成註冊自動登入；重整後維持登入狀態；登出後無法存取受保護頁面。

---

**5.2 目標優先級設定（新增）**
- 每個 Objective 新增「優先級」欄位，選項為 1 / 2 / 3，預設為 2
- 優先級顯示在 OKR 列表頁每個 Objective 旁，可直接點擊修改，不需進入編輯頁
- 係數對應：優先級 1 → 係數 1，優先級 2 → 係數 2，優先級 3 → 係數 3
- **驗收標準**：
  - 建立 Objective 時可設定優先級，預設為 2
  - OKR 列表頁可直接修改優先級，修改後 Dashboard 的 Idea 排序即時重新計算
  - 所有目標優先級相同時，行為等同簡單平均，不出錯

---

**5.3 Dashboard 重新設計：Idea 優先排行榜（核心改版）**

取代現有的「目標進度快照 + Idea 列表」結構，Dashboard 主角改為按加權貢獻分數排序的 Idea 清單。

加權分數計算公式：
```
加權分數 = Σ(目標貢獻分 × 目標優先係數) / Σ(目標優先係數)
```

畫面結構：
- 每個 Idea 顯示：標題、加權分數、展開鍵（`›`）
- 點擊展開鍵後，顯示一句 AI 理由，說明為何此 Idea 現在得分最高（帶入目標進度脈絡）
- 未分析的 Idea 排在最後，顯示「待評估」標籤
- Task 狀態為 done 的 Idea 視覺上降調（灰色），排在清單底部

**驗收標準**：
- Dashboard 載入後，所有已分析 Idea 依加權分數由高到低排列
- 修改任一目標優先級後，排序在 1 秒內重新計算並更新
- 理由預設收合，點擊後展開，再點擊收合，畫面不跳動
- 完成 Task 後，該 Idea 移至清單底部，不需重整頁面

---

**5.4 Idea 輸入簡化（降低摩擦）**

現有流程要求用戶填寫標題、為什麼要做、預期成效、備註，再點擊分析。改版後：

- **快速輸入模式**：使用者只需輸入 Idea 標題，點擊「快速評估」，AI 立刻根據標題給出初步加權分數與一句理由
- 「為什麼要做」、「預期成效」、「備註」三個欄位改為**選填**，折疊收起，標示「補充說明（選填，有助提升分析精準度）」
- 補充欄位填寫後，可點擊「重新分析」獲得更精準的評分
- **驗收標準**：
  - 只填標題即可完成分析並儲存 Idea
  - 快速評估結果在 3 秒內顯示
  - 有補充說明的分析結果，理由明顯比純標題分析更具體（QA 人工驗收）

---

**5.5 Idea 分析理由帶入目標進度脈絡（理由品質提升）**

現有的 AI 理由是通用貢獻分析。改版後，`analyzeIdea` 的 prompt 需加入目標當前進度資訊，讓理由反映使用者的真實狀態。

理由品質目標：
- 舊：「這件事與你的健康目標高度相關。」
- 新：「你的健康目標本月進度僅 15%，這件事直接推進核心 KR，現在是效益最高的行動。」

**驗收標準**：
- 理由中應包含：目標名稱、目標當前完成度（若有 KR 進度資料）、為何此時機效益高
- 當目標尚無任何 KR 進度時，理由改為說明貢獻路徑，不強行帶入進度數字

---

**5.6 OKR 目標建立（維持 v1 核心，小幅調整）**
- 維持現有 AI 自動分類 KR 類型的邏輯（`classifyKR`）
- 新增：建立 Objective 時即可設定優先級（見 5.2）
- 移除：不在建立流程開頭加入引導式問答（AI 教練介入時機改為 Idea 輸入時）
- **驗收標準**：同 v1，另加「建立完成後 Objective 優先級正確顯示在列表」

---

**5.7 KR 進度追蹤**（同 v1，無功能變更）
- 累積型：Task 完成自動累加 / 手動 Check-In
- 測量型：手動 Check-In 填絕對值
- 里程碑型：完成即 100%
- **驗收標準**：同 v1

---

**5.8 Task 管理**（同 v1，新增 Dashboard 聯動）
- 維持 Idea → Task 升級、todo / in-progress / done 三種狀態
- 新增：Task 完成後，Dashboard 排行榜即時將該 Idea 移至底部
- **驗收標準**：同 v1，另加「Task 完成後 Dashboard 排序即時更新，無需重整」

---

### Nice-to-Have — P1

**5.9 AI 教練：模糊 Idea 單問介入（新增）**

觸發條件：使用者送出 Idea 分析後，若 AI 判斷該 Idea 與所有目標的相關性均 <3 分（極低相關），在顯示分數前先詢問一個釐清問題。

範例：
- 輸入「提升自己」→ AI 問：「你說想提升自己，是指哪個方向——工作技能、身體健康，還是其他？這會影響我怎麼評估這件事的優先級。」
- 用戶回答後，AI 重新評分並在理由中說明前後差異

規則：
- 最多問一個問題，不做多輪引導
- 用戶可略過（點「跳過，直接評分」），以現有標題繼續分析
- 只在相關性極低時觸發，高分 Idea 不詢問

**驗收標準**：
- 模糊 Idea 觸發問題的比例合理（目標：相關性均 <3 的 Idea 中，≥80% 觸發詢問）
- 用戶回答後的重新評分，理由比原始分析更具體
- 跳過功能正常運作，不阻擋儲存流程

---

**5.10 分數動態更新（KR 進度影響排序）**

當用戶更新 KR 進度後，相關 Idea 的加權分數理由自動標記為「需重新評估」（視覺提示），使用者可點擊「重新評估」觸發重新分析，而非系統自動靜默重算（避免用戶感覺排序莫名變動）。

**驗收標準**：
- KR 進度更新後，關聯 Idea 旁出現「進度已更新，重新評估？」提示
- 點擊後觸發重新分析，新理由帶入最新進度數據
- 未點擊時舊分數保持不變，不影響排序

---

**5.11 AI 輔助 OKR 精煉（Guided 流程）**（同 v1 P1，優先級不變）
- 整合現有 `refineObjective` → `suggestKeyResults` → `generateSnapshot` 三階段 API 為完整引導式 UI
- 入口：OKR 建立頁的「AI 幫我想」按鈕

---

**5.12 KR 信心度標記與 AI 建議**（同 v1 P1，優先級不變）
- 在 KR 編輯頁加入 on-track / at-risk / needs-rethink 標記 UI
- 標記後 AI 自動給出分析建議（`analyzeConfidenceDrop` API 已存在）

---

**5.13 季度評分（Quarter Scoring）**（同 v1 P1，優先級不變）
- 季末流程 UI：為每個 KR 輸入 0.0–1.0 分數，AI 給出 continue / complete / reset 建議
- `getQuarterRecommendation` API 已存在，補充 UI 頁面

---

### Future Considerations — P2

**5.14 目標優先級自動推薦**：AI 根據目標的當前進度、活躍度與設定時間，主動建議用戶調整優先級係數。

**5.15 週報摘要**：每週自動生成一份「本週完成了哪些行動、哪些目標有進展、下週最值得做什麼」的摘要推送。

**5.16 通知與提醒、匯出報告、團隊協作、原生 App**：同 v1 P2，維持不變。

## 6. 成功指標（Success Metrics）

### 領先指標（上線後 1–4 週）

| 指標 | 目標 | 衡量方式 |
|------|------|---------|
| Dashboard 打開後點擊「展開理由」的比例 | ≥50% | 事件追蹤 |
| 快速輸入模式使用率（只填標題就送出分析） | ≥60% | 比較有/無補充說明的分析請求比例 |
| Idea 排序後執行第一名 Idea 的比例（Task 狀態變為 in-progress） | ≥40% | Task 狀態變更追蹤 |
| 目標優先級設定率（至少一個目標非預設值 2） | ≥30%（30 天內） | 資料庫查詢 |
| 新用戶從首次輸入到第一個 Idea 分析完成的時間 | <3 分鐘 | Session 時間追蹤 |

### 落後指標（上線後 4–12 週）

| 指標 | 目標 |
|------|------|
| D28 留存率 | ≥30% |
| 每位活躍用戶每週平均打開 App 次數 | ≥3 次（目標：30 秒決策習慣） |
| 有至少一個 Task 完成並更新 KR 進度的用戶比例 | ≥50%（30 天內） |
| NPS 分數 | ≥40 |

## 8. 時程考量（Timeline Considerations）

### Phase 1 — 核心循環清晰化（現在）
重新設計 Dashboard（5.3）、目標優先級設定（5.2）、Idea 輸入簡化（5.4）、理由品質提升（5.5）。解決 Q1、Q2 安全阻塞。

**自我驗收標準**：自己連續使用一週，每次打開 App 後 10 秒內明確知道今天該做什麼，且理由說服力足夠讓你去執行。

### Phase 2 — 動態回饋與 AI 教練（4–6 週後）
加入分數動態更新提示（5.10）、模糊 Idea 單問介入（5.9）、AI 引導式 OKR 建立（5.11）。

### Phase 3 — 回顧與深化（3 個月後）
季度評分 UI（5.13）、KR 信心度標記（5.12），根據留存率數據決定是否進入 P2。

## 9. 系統架構概覽（Architecture Overview）

同 v1，新增說明：

- `objectives` 表的 `meta` JSONB 欄位新增 `priority` 屬性（1 / 2 / 3）
- `analyzeIdea` 的 prompt 需加入 KR 當前進度資訊（`currentValue`、`targetValue`、完成百分比）
- Dashboard 的加權分數計算在前端即時進行，不需額外 API 呼叫

### 資料模型更新

| 資料表 | 變更 |
|--------|------|
| `objectives` | `meta` JSONB 新增 `priority: 1 \| 2 \| 3`，預設 2 |
| `ideas` | 新增 `quick_analysis: boolean`，標記是否為快速評估（僅標題分析） |
